# Triton-Ascend Performance Analysis Methods


## Obtaining Performance Data


Before performing performance optimization, it is necessary to obtain accurate performance data, understand the current performance status, and analyze the next optimization direction based on the performance status. MindStudio provides various methods for testing Triton operator performance, including on-board profiling and single-operator performance simulation pipeline diagrams.


### Onboard Profiling


The msProf tool is used to collect and analyze key performance indicators of operators running on the Ascend AI processor. Users can quickly locate software and hardware performance bottlenecks of operators based on the output performance data, thereby improving the efficiency of operator performance analysis.


- Note: The use of the msProf tool depends on the msopprof executable file in the CANN package. The interface functions in this file are consistent with msprof op. This file comes with the CANN package and does not require separate installation. For common commands of the msProf tool, please refer to [msProf Common Commands](https://www.hiascend.com/document/detail/zh/mindstudio/82RC1/ODtools/Operatordevelopmenttools/atlasopdev_16_0082.html).


The following command provides an example of collecting performance data for an operator on the board. You can flexibly combine and configure parameters according to your own needs. In the example, --output is an optional parameter used to specify the storage path for the collected performance data; --kernel-name is an optional parameter used to specify the performance data of a single kernel to be collected (if not specified, only the first operator scheduled during program execution will be collected); $HOME/projects/test_op.py is the operator executable script.


```python
msprof op --kernel-name=target_kernel_name --output=$HOME/projects/output python3 $HOME/projects/test_op.py
```


Take the demonstration test case [03-layer-norm.py](../../../third_party/ascend/unittest/autotune_ut/03-layer-norm.py) as an example (when `--output` is not specified, the generated data files are saved in the current path):


```python
msprof op --kernel-name=_layer_norm_fwd_fused python3 03-layer-norm.py
```


- Note: For the meaning of the result data of all the following collection items, refer to the [op_summary (Operator Details)](https://www.hiascend.com/document/detail/zh/canncommercial/83RC1/devaids/Profiling/atlasprofiling_16_0067.html) section in the *CANN Performance Tuning Tool User Guide*.

#### Simulation Pipeline Collection of Triton Operator Debug Version


The simulation trace (trace.json / visualize_data.bin) by default only contains instruction addresses and instruction names. To use the instruction correlation panel in MindStudio Insight and view the Triton/Python source lines and call stacks corresponding to instructions (i.e., Method 4 in [Finding Bottlenecks](#finding-bottlenecks)), the operator must be compiled with debug_line debugging information enabled. When this is not enabled, msprof op simulator will output the following prompt, and the code correlation file will be empty:


```text
[WARN] Kernel missed debug_line information. If you need code call stack, please recompile kernel with -g option
[WARN] Code call stack is empty
[WARN] Lack of code info of files
```


Triton operators do not require manually adding `-g`. The triton-ascend backend controls whether to append `--enable-debug-info=true` to the `bishengir-compile` command via the environment variable `TRITON_DISABLE_LINE_INFO`. Note that the default value of triton-ascend is `true` (i.e., line number information is **disabled** by default, contrary to the community Triton default of enabled), and it must be explicitly set to `False`:


```bash
export TRITON_DISABLE_LINE_INFO=false
```


Verify whether it is effective: The `[DEBUG] cmd_list:` line in the log should contain `--enable-debug-info=true`, and the `Kernel missed debug_line information` warning should disappear. Then import `visualize_data.bin` into MindStudio Insight to view the source code line and call stack associated with each instruction in the instruction timing diagram.


## Analyzing Performance Data


### Theoretical Parameters


Theoretical performance represents the ideal target for actual operator performance. Different hardware platforms have varying hardware specifications, and theoretical performance helps us understand the hardware's potential, thereby setting performance optimization goals.


The theoretical time for data movement-related pipelines (MTE1/MTE2/MTE3, etc.) = amount of data moved (unit: Byte) / theoretical bandwidth. For example, the GM peak bandwidth of a certain AI processor is approximately 1.8 TB/s. To move a matrix of float data type with a size of 4096 * 4096, the theoretical time for the movement is `sizeof(float) * 4096 * 4096 / 1.8 TB/s = 37.28 us` (calculated based on 1 TB = 10<sup>12</sup> Byte).


> Note:
>
> - When multiple copy instructions exist simultaneously, bandwidth sharing may occur, and each instruction cannot copy data at a rate close to the theoretical bandwidth. For example, when MTE2/MTE3 perform GM read and write operations simultaneously, the time consumed by the copy pipeline should be (MTE2 copy volume + MTE3 copy volume) / GM bandwidth.
> - When copying data blocks of different sizes, the bandwidth utilization (effective bandwidth / theoretical bandwidth) varies. For cases where the amount of data copied per operation is small, the measured performance may not reach the theoretical bandwidth.
>
- Theoretical time consumption for computation-related pipelines (Cube/Vector/Scalar, etc.) = amount of computation data (unit: Element) / theoretical computing power. For example: the theoretical peak computing power of a certain AI processor for Vector operations on float data type is 11.06 TOPS. To perform a single instruction computation on 32K Elements of float type, the theoretical time consumption is 32K / 11.06 TOPS = 0.003us (calculated with 1K = 1000).


### Finding Bottlenecks


After obtaining performance data, areas with significant deviations from theoretical values and processes with long execution times are considered "bottlenecks." The following section will explain how to identify bottlenecks through performance data and determine corresponding optimization directions.


- Method 1: Analyze pipeline status through on-board Profiling \
View the op_summary_*.csv file parsed from on-board Profiling to analyze pipeline status. Note: "*" indicates a timestamp.
![analyse_data_op_summary](../figures/performance_analysis_analyse_data_op_summary.png)


The utilization rate of each pipeline should ideally be 100%. Any pipeline that does not reach 100% may have room for improvement. In the example above, data obtained from a certain AI processor shows that in the first scenario of the Vector operator `_layer_norm_fwd_fused`, the Vector pipeline utilization rate `aiv_vec_ratio` is less than 10%, indicating that the computing power is not fully utilized; the Scalar pipeline utilization rate `aiv_scalar_ratio` is already around 60%, suggesting that Scalar is the longest pipeline.  

When Scalar is the longest pipeline: it is necessary to analyze whether the operator source code involves complex operations on scalar values. Ascend's SIMD microarchitecture is more suitable for multi-data parallel computing. Another possibility is that, due to some instructions not supporting specific data types in hardware, the Triton software stack degenerates vector computations into scalar computations. Optimization should be combined with pipeline and scalar optimization methods. Refer to Method 3, viewing the simulation pipeline diagram, and Method 4, checking code hotspots for further analysis.  

For more general cases, such as MTE2 data movement and actual scenarios: the shapes of the three input matrices are (128,128), (128,1), and (128,1), with the data type being float16. The current algorithm uses the Two-pass method, so there are three loads of X and one load each of W and B. From this, the total amount of data to be moved can be calculated. Then, using the theoretical calculation method for pipeline movement time introduced in [Theoretical Parameters](#theoretical-parameters), the theoretical value is `sizeof(float16) * (128 * 128 * 3 + 128 + 128) / 1.8 TB/s ≈ 0.1991 us` (calculated based on 1 TB = $10^{12}$ Bytes), which shows a significant gap from the actual performance data `aiv_mte2_time`. Analysis reveals that the total size of the input data is smaller than the UB space (192KB for the A2 model). Therefore, the excessively long MTE2 time may be due to the basic blocks calculated by Tiling being too small, resulting in redundant movement instructions being issued. Optimization should combine pipeline optimization and Tiling optimization methods. Refer to Method 3, viewing the simulation pipeline diagram, to further analyze the status of each pipeline.


- Method 2: Analyze Tiling Situation through On-Board Profiling \
The AI processor used in the previous example can be observed through the hardware platform to have 48 Vector cores. The _layer_norm_fwd_fused operator is a pure Vector operator, but in some scenarios, too many Blocks are dispatched (Block Dim > 48), causing excessive Host scheduling overhead. Therefore, the next main optimization direction is Tiling optimization.


- Method 3: Analyze pipeline status through simulation pipeline diagram \
![analyse_data_waveform](../figures/performance_analysis_analyse_data_waveform.png) \
    The example above shows data obtained from a certain AI processor simulator. It can be observed that the SCALAR and FLOWCTRL instructions of the Vector core are saturated. Combined with operator logic analysis, it is necessary to determine whether there are excessive scalar computations and unsupported vectorization operations. The next main optimization direction is scalar computation optimization. On the other hand, it can be seen that the relevant pipelines of the Vector core (such as MTE2 and VECTOR of veccore0) exhibit periodic pipeline stalls, i.e., large blank segments with no operations. Combined with operator logic analysis, it is necessary to analyze whether factors such as overly small basic block partitioning cause these stalls. The main optimization direction is pipeline optimization, followed by Tiling optimization and memory optimization to further improve Vector pipeline utilization.


- Method 4: Analysis via Code Hotspot Profiling \
![analyse_data_code_mapping](../figures/performance_analysis_analyse_data_code_mapping.png) \
    The example above shows data obtained from a certain AI processor simulator. It can be seen that the load interface on the left corresponds to a set of assembly instructions on the right (filtered to display only code-line-related instructions, sorted in descending order by execution cycles). The proportion of scalar instructions is high, which does not match the expected scenario where load, as a memory access interface, should have a high MTE proportion. Therefore, the main optimization direction is scalar computation.


### Example: i64/i32 compare cannot enable vector on NPU, causing vector computation to fall back to scalar computation


[Description] The cmp of i64/i32 cannot enable vector on NPU, degrading to scalar computation with reduced efficiency; by converting to fp32, vec_cast and vec_cmp can be used to implement vector operation acceleration.  
[Note] When using cmp in mask within tl.load and tl.save, the compiler can automatically optimize to vec operations in most cases. In this example, tl.where requires manual conversion.


```diff
import triton
import triton.language as tl

@triton.jit
def npu_vector_cmp_kernel(
    X,                 # [Tensor] input tensor (row x col)
    Out,               # [Tensor] output tensor (row x col)
    Mean,              # [Vector] mean tensor (row, ) of X
    Rstd,              # [Vector] std tensor (row, ) of X
    stride_x_row,      # [Scalar] stride of row in X
    stride_out_row,    # [Scalar] stride of row in Out, normally equals to stride_x_row
    M,                 # [Scalar] row number
    N,                 # [Scalar] col number
    eps,               # [Scalar] epsilon to avoid division by zeros
    BLOCK_M: tl.constexpr,
    BLOCK_N: tl.constexpr
):
    """
    an example of layernorm to checkout Vector Cmp
    Out = ((X - E[X]) / sqrt(V[X] + eps)) on dim -1

    just for easy case, we assume that:
    1. BLOCK_N >= X.shape(-1), group_n = 0 only
    2. BLOCK_M = 1, group_m = range(0, row, 1)
    """
    group_m = tl.program_id(0)
    group_n = tl.program_id(1)
    row = group_m

    # calculate index & offset
    Mean = Mean + group_n * M
    Rstd = Rstd + group_n * M
    X = X + row * stride_x_row + group_n * N
    Out = Out + row * stride_out_row + group_n * N

    cols = tl.arange(0, BLOCK_N)  # cols is int64
    x = tl.load(X + cols, mask=cols < N, other=0.0).to(tl.float32)

    # calculate mean & rstd
    mean = tl.sum(x, axis=0) / N
    tl.store(Mean + row, mean)

-   xbar = tl.where(cols < N, x - mean, 0.0) # N为标量值

+   # change cols(i64) into cols_cmp(f32) to enable vector processing
+   cols_cmp = cols.to(tl.float32)
+   xbar = tl.where(cols_cmp < N, x - mean, 0.0)

    var = tl.sum(xbar * xbar, axis=0) / N
    rstd = 1 / tl.sqrt(var + eps)
    tl.store(Rstd + row, rstd)

    # calculate Out
    mask = cols < N
    out = (x - mean) * rstd
    tl.store(Out + cols, out, mask=mask)
```
