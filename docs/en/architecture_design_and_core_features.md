# Architecture Design and Core Features


## 1. Logical Architecture


![Image](./figures/architectural_diagram.png)
**Triton-Ascend Architecture Description**


**Core Components:**


- **`Ascend language extension`**: Triton language extension adapted for Ascend
- **`compiler`**: Triton compiler adapted for Ascend
- **`driver`**: Device driver interface adapted for Ascend


**Component Functions:**


- **`Ascend language extension`**
  Introduces syntax and semantic extensions tailored for the Ascend NPU architecture based on the standard Triton language.


- **`compiler`**
  Receives the intermediate representation file `TTIR` (Triton IR) generated by the upper-layer Triton compiler, and performs a series of transformations adapted to Ascend hardware.


  ```python
  Triton IR → Linalg IR → AscendNPU IR → triton_xxx_kernel.o
  ```


Triton IR is converted to Linalg IR, and then through BiSheng Compiler, an executable binary file `triton_xxx_kernel.o` targeting Ascend NPU is generated.


- **`driver`**
  Provides the interface capability between the Triton runtime and the Ascend software stack (CANN), loading the device-side executable kernel `triton_xxx_kernel.o` generated by the BiSheng Compiler.


## 2. Code Structure


### 2.1 Code Structure Principles


This project extends the standard Triton to support Huawei Ascend NPU (via the CANN software stack). The overall design follows the following **code principles**:


> - **If the modification is target independent**, it should remain in the **Triton core** section (e.g., general modifications to language, runtime);
> - **If the modification is strongly related to Ascend hardware** (target affinitive), it should be placed in **Triton-Ascend**.


### 2.2 Directory Structure and Functional Description


| Directory or File | Corresponding Architecture Layer | Functional Description |
| --- | --- | --- |
| `python/` | Triton core | Retains the general Python-side implementation of standard Triton, including `triton.language`, JIT, runtime, caching, toolchain entry points, etc. Hardware-independent general capabilities are prioritized in this directory. |
| `include/` and `lib/` | Triton core | Retains the general C++/MLIR infrastructure, Dialect, Pass, and conversion logic of standard Triton. Ascend-specific backend implementations are not hosted here. |
| `third_party/ascend/` | Triton-Ascend | Root directory of the Ascend backend, centrally hosting language extensions, compilation backend, runtime driver, MLIR Pass, examples, and tests strongly related to Ascend NPU, CANN, and BiSheng Compiler. |
| `third_party/ascend/language/` | Ascend language extension | Ascend language extension directory; after installation, it is linked under `triton.language.extra` for use by Triton kernels via `triton.language.extra.cann`. |
| `third_party/ascend/language/cann/libdevice.py` | Ascend language extension | Python interface of `libdevice` adapted for Ascend NPU, providing mathematical functions and low-level operator wrappers for invocation by Triton operators. |
| `third_party/ascend/backend/compiler.py` | compiler | Main entry point of the Ascend compiler backend, responsible for registering compilation options, orchestrating the conversion from TTIR to Ascend-adapted IR, Linalg/LLVM stages, and invoking subsequent toolchains to generate executable binaries. |
| `third_party/ascend/backend/driver.py` | driver | Ascend runtime driver module, responsible for interfacing with runtime environments such as CANN/torch_npu, loading and launching compiled device-side executables. |
| `third_party/ascend/include/` and `third_party/ascend/lib/` | compiler | Ascend-specific MLIR Dialect, Pass, and conversion implementations, such as `TritonToLinalg`, `TritonToStructured`, `DynamicCVPipeline`, `AutoBlockify`, etc. |
| `third_party/ascend/AscendNPU-IR/` | compiler | Ascend NPU-related IR and BiSheng compilation chain adaptation content, an important component that continues from the Triton-Ascend compilation flow down to hardware-side code generation. |
| `third_party/ascend/tutorials/` and `third_party/ascend/unittest/` | Examples and Tests | Provides Triton examples, migration samples, Python unit tests, and MLIR conversion tests on the Ascend platform, used to verify the capabilities of the Ascend backend. |

##### 3.2.2.1 TritonToStructured


Process integer division and modulo operations in pointer expressions and mask expressions, and regenerate operations such as load/store after removing the integer division and modulo through dimension elevation.


| Converter                | Function | Limitations |
| ------------------------ | -------------------------- | ------------------------- |
| RewriteAddPtrOp          | Analyzes pointer expressions (`AddPtrOp`) in operations such as `tl.load`, `tl.store`. Decomposes and models the original pointer offset calculation into a `PtrState` object containing specific offset information for each dimension (axis). For example, for an expression like `ptr + x // 1024 * 4096 + x % 1024 * 4 + y`, it analyzes the contributions and relationships of the `x` and `y` axes. | 1. The original iteration axis involved (e.g., `x`) must be divisible by the split axis (e.g., `1024`).<br>2. The external `XBLOCK` size must be an integer multiple or a divisor of the split axis `divisor`. |
| CreateAddpr              | Based on the analyzed `PtrState` object, reconstructs a new `AddPtrOp` pointer calculation operation. The newly generated pointer expression eliminates the integer division (`//`) and modulo (`%`) operations from the original expression. | Depends on a valid `PtrState` successfully generated by `RewriteAddPtrOp`. |
| RewriteLoadOp            | Analyzes the mask expression in `tl.load` operations. Decomposes and models complex mask conditions containing division/modulo into a `MaskState` object containing boundary information for each dimension. For example, for `mask = x // 1024 < 8 and x % 1024 < 1024 and y < 4`, it analyzes the independent constraint conditions for each dimension. | 1. The original iteration axis involved (e.g., `x`) must be divisible by the split axis (e.g., `1024`).<br>2. The external `XBLOCK` size must be an integer multiple or a divisor of the split axis `divisor`. |
| BuildMask                | Based on the analyzed `MaskState` object, reconstructs a new mask expression. The new mask eliminates the integer division (`//`) and modulo (`%`) operations from the original expression. | Only processes `MaskState` generated by `RewriteLoadOp` or `RewriteStoreOp`. Cannot handle arbitrarily complex, non-canonical mask expressions. |
| CreateLoad               | Uses the new pointer expression generated by `CreateAddpr` and the new mask expression generated by `BuildMask` to recreate (replace) the original `tl.load` operation, completing the instruction rewrite. | Depends on the successful execution of prerequisite steps such as `RewriteAddPtrOp`, `CreateAddpr`, `RewriteLoadOp`, and `BuildMask`. |
| RewriteStoreOp           | Analyzes the mask expression in `tl.store` operations. Its function is similar to `RewriteLoadOp`, decomposing and modeling complex mask conditions containing division/modulo into a `MaskState` object. | Same as `RewriteLoadOp`. |
| CreateStore              | Uses the new pointer expression generated by `CreateAddpr` and the new mask expression generated by `BuildMask` to recreate (replace) the original `tl.store` operation, completing the instruction rewrite. | Depends on the successful execution of prerequisite steps such as `RewriteAddPtrOp`, `CreateAddpr`, `RewriteStoreOp`, and `BuildMask`. |
| RewriteAtomicRWMOp       | Handles pointer issues in atomic read-write-modify operations (e.g., `atomic.add`, `atomic.max`). | Generally inherits the same limitations as `RewriteAddPtrOp`. May not support certain special, non-contiguous, or conditional atomic operation patterns. |
| RewriteAtomicCASOp       | Handles pointer linearization issues in atomic compare-and-swap operations (`atomic.cas`). Analyzes the pointer expression and eliminates division and modulo operations through dimension lifting to match the addressing requirements of hardware atomic instructions. | |
| RewriteWhile             | Handles pointer accumulation operations within `while` loop bodies. | Does not support complex pointer path transformations involving conditional branches (`if`) within the loop body. |
| RewriteFor               | Handles pointer accumulation operations within `for` loop bodies. | |


##### 3.2.2.2 TritonToUnstructured


| No. | Pass / Converter                              | Description                                                                                                                                                                                                                             |
|-----|---------------------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| 1   | discrete-mask-access-conversion             | Analyzes and converts memory access patterns in Triton based on discrete index masks (e.g., `triton.language.load` with non-contiguous masks), preparing for the subsequent expansion of discrete axes into loops. This pass identifies irregular or sparse access patterns that cannot be efficiently handled by backend hardware. |
| 2   | triton-to-unstructured                      | Converts tensor operations containing discrete axes, identified by `discrete-mask-access-conversion`, into scalar memory accesses based on explicit scalar loops.                                                                         |
| 3   | bubble-up-operation                         | Primarily optimizes the upward movement of `extract op/extract_slice` sequences. This improves data locality and, in some cases, eliminates unnecessary loops generated after conversion, thereby enhancing the execution efficiency of the generated code. |


###### 3.2.2.2.1 discrete-mask-access-conversion


| Converter Name                | Description                                                                                                                                                                                                                             |
|------------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| DiscreteMaskStoreConversion  | First, perform mask analysis. If the mask analysis result is non-contiguous, convert the original store operation into the following sequence:<br>1. load (load the content of the target storage address)<br>2. select (select the target storage content and the value content to be stored based on the mask)<br>3. store (store the selected result back to the target address) |
| DiscreteMaskLoadConversion   | First, perform mask analysis. If the mask analysis result is non-contiguous, convert the original load operation into the following sequence:<br>1. load (load all content of the source tensor)<br>2. select (select the source tensor content based on the mask, setting the masked part to the other value) |
| DiscreteMaskAtomicAddConversion | First, perform mask analysis. If the mask analysis result is non-contiguous, convert the original atomic_add operation into the following sequence:<br>1. select (select the value based on the mask, setting the masked part to 0)<br>2. atomic_add (regenerate the atomic_add operation using the selected result) |
