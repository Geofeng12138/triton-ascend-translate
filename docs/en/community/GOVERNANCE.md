# Governance Mechanism


## Mission


As an important part of the Triton ecosystem, the Triton-Ascend project is committed to providing all Ascend NPU users with an efficient and easy-to-use Triton compilation framework, enabling Triton code to run efficiently on Ascend hardware, and actively contributing to enriching the Triton and Ascend ecosystem.


## Principles


- **Open and Transparent**: All technical decisions and community discussions are conducted through public channels.
- **Contribution-Oriented**: Membership is based on individual ability and contributions, rather than being tied to an employer.
- **Community-Driven**: Broad participation is encouraged, diverse perspectives are respected, and consensus is reached through discussion and voting.


## Governance Structure


Triton-Ascend is an open-source project under the Triton community, and role appointments are ultimately determined by the Triton community. The project adopts a hierarchical technical governance structure:


### Contributors


**Responsibilities**:


- Help new contributors integrate into the project
- Handle and respond to community `Issue`
- Review `RFC` and code


**Requirements**:


- Complete at least one contribution  
- Continuously and actively participate in the project by submitting `Issues`, conducting code reviews, making code commits, and engaging in community activities


**Permissions**: Grant the `Triage` role for the GitHub repository, including read and clone permissions, as well as the ability to submit Issues and PRs.


### Maintainer


**Responsibilities**:


- Define the project's vision and mission, shape the technical direction, and ensure long-term success  
- Have code merge permissions and lead roadmap planning  
- Review community contributions and continuously improve code  
- Be responsible for code quality and architectural decisions of core modules  
- Participate in release decisions  
- Guide contributors through code contributions  
- Participate in community technical meetings to discuss and vote on technical direction, major proposals, and `Release` matters


**Requirements**:


- Deep understanding of the Triton and Triton-Ascend codebases  
- Commitment to ongoing code contributions, with the ability to design, develop, and review PRs  
- Active participation in community code reviews, delivery of at least one major feature, and sustained high-quality contributions  
- Proactive handling of `Issue`, responding to inquiries, and engaging in discussions  
- Significant technical contributions to the project (e.g., core feature development, architecture design, performance optimization)  
- Demonstrated strong technical judgment and community collaboration skills


**Permissions**: Grant write access to the GitHub repository, including permissions to read, clone, and push code, core branch protection and merge permissions, release execution permissions, as well as permissions to manage `Issue` and `Pull Request`.


Promotion is jointly reviewed by existing `Maintainer`s, taking into account contribution quality, continuity, and community engagement.


## Nomination and Removal


### Principles


- Membership is based on individual ability, demonstrated through contributions, reviews, and discussions
- Candidates must align with the principles of Triton-Ascend and maintain consistency in their actions
- A `Maintainer` who has not contributed to the project for **6 consecutive months** may be transitioned to an honorary role
- Membership is granted to individuals and is not tied to their employer


### Nomination Process


- Anyone can initiate a nomination, including self-nomination, through a community `Issue`.
- The nomination must include a record of the candidate's contributions and a description of their technical capabilities.
- The existing `Maintainer` reviews and evaluates the nomination, requiring **approval from more than half of the current Maintainers with no objections**.


### Regular Review


- **Monthly** review of all `Maintainer` activity during technical meetings
- A `Maintainer` who has not contributed to the project for **6 consecutive months** may be transitioned to honorary status after discussion
- An `Honorary` `Maintainer` may apply to restore `Maintainer` status at any time by resuming active contributions
