# Governance Mechanism


## Mission


As an important part of the Triton ecosystem, the Triton-Ascend project is committed to providing all Ascend NPU users with an efficient and easy-to-use Triton compilation framework, enabling Triton code to run efficiently on Ascend hardware, and actively contributing to enriching the Triton and Ascend ecosystems.


## Principles


- **Open and Transparent**: All technical decisions and community discussions are conducted through public channels.
- **Contribution-Oriented**: Membership is based on individual ability and contributions, rather than affiliation with an employer.
- **Community-Driven**: Broad participation is encouraged, diverse perspectives are respected, and consensus is reached through discussion and voting.


## Governance Structure


Triton-Ascend is an open-source project under the Triton community, and role appointments are ultimately determined by the Triton community. The project adopts a layered technical governance structure:


### Contributors


**Responsibilities**:


- Help new contributors integrate into the project
- Handle and respond to community `Issues`
- Review `RFCs` and code


**Requirements**:


- Complete at least one contribution
- Actively and continuously participate in the project through submitting `Issue`s, code reviews, code commits, and engaging in community activities


**Permissions**: Grant the `Triage` role for the GitHub repository, including read and clone permissions, as well as the ability to submit Issues and PRs.


### Maintainer


**Responsibilities**:


- Define the project's vision and mission, shape the technical direction, and ensure long-term success  
- Possess code merge permissions and lead roadmap planning  
- Review community contributions and continuously improve code  
- Be responsible for code quality and architectural decisions of core modules  
- Participate in release decisions  
- Guide contributors through code contributions  
- Participate in community technical meetings to discuss and vote on technical direction, major proposals, and `Release` matters


**Requirements**:


- Deep understanding of the Triton and Triton-Ascend codebases
- Committed to continuous code contributions, with the ability to design, develop, and participate in the PR review process
- Actively involved in community code reviews, having delivered at least one major feature, and maintaining sustained high-quality contributions
- Proactively handles `Issue`s, responds to inquiries, and participates in discussions
- Made significant technical contributions to the project (e.g., core feature development, architecture design, performance optimization, etc.)
- Demonstrates strong technical judgment and community collaboration skills


**Permissions**: Grant write access to the GitHub repository, including permissions to read, clone, and push code, core branch protection and merge permissions, release execution permissions, as well as permissions to manage `Issue` and `Pull Request`.


Promotion is jointly reviewed by existing `Maintainer`s, taking into account contribution quality, continuity, and community engagement.


## Nomination and Removal


### Principles


- Membership is based on individual capability, demonstrated through contributions, reviews, and discussions.
- Candidates must align with the principles of Triton-Ascend and maintain consistency in their behavior.
- A `Maintainer` who has not contributed to the project for **6 consecutive months** may be transitioned to an honorary status.
- Membership is granted to individuals and is not tied to their employer.


### Nomination Process


- Anyone can initiate a nomination, including self-nomination, through a community `Issue`.
- The nomination must include the candidate's contribution record and a description of their technical capabilities.
- Existing `Maintainers` review and evaluate the nomination, requiring **more than half of the current Maintainers to agree with no objections**.


### Regular Review


- **Monthly** review of all `Maintainer` activity during technical meetings
- A `Maintainer` who has not contributed to the project for **6 consecutive months** may be transitioned to an honorary status after discussion
- An `Honorary` `Maintainer` may apply to restore their `Maintainer` status at any time by resuming active contributions
